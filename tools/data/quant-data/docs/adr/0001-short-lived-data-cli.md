# Short-Lived Data CLI Owns Persistent Maintenance State

Status: accepted

finance-research tools integrate with the data-maintenance CLI through short-lived JSON commands rather than a managed daemon process. The CLI owns persistent external data, provider configuration, maintenance queue state, provider limit state, and failure state under the user's local quant-data directory; each command advances eligible maintenance work before returning so background updates can progress without an always-on worker. Tool-facing commands use `quant-data <method>` with kebab-case method names, camelCase JSON input/output, and a unified JSON envelope that separates returned data from provider and maintenance status; interactive reads use read-through semantics, returning refreshed data when possible and cached data with maintenance error codes/messages when providers are limited or unavailable. Concurrent CLI processes block while waiting for store and maintenance locks, but calls support a deadline with a default of 10 seconds; background maintenance advances best-effort within that deadline and leaves unfinished work in persistent status for later commands. Consumer state comes from command envelopes and polling `quant-data status`, not push events. Exit code zero means the CLI produced a valid envelope, while maintenance errors live inside that envelope; nonzero exits are reserved for process, protocol, or stdout failures. This favors a simpler process lifecycle and inspectable command contracts over push-style status events and in-memory scheduling.

Implementation note: the current Go CLI does not accept a command JSON `deadline` field. Callers enforce process deadlines around `quant-data`, while store waits and provider calls are bounded internally where implemented.
